Project brief

What is Econ Trends?

A reproducible research workflow for tracking whether top economics journal articles are framed around causal questions, predictive questions, or neither.

Scope

Five general-interest journals

American Economic Review, Quarterly Journal of Economics, Journal of Political Economy, Econometrica, and Review of Economic Studies.

Period

1975-2025 article panel

The current working dataset covers publication years 1975 through 2025, with a closer first-batch inspection of 2023-2025.

Question

How research framing changes

The dashboard reports descriptive shifts in the share of articles classified as causal, predictive, other, or insufficient text.

Causal Estimating, identifying, or interpreting causal effects.
Predictive Prediction, forecasting, classification, performance, or out-of-sample accuracy.
Other Papers that do not clearly fit the causal or predictive definitions.
Insufficient text Records where available title/abstract/full-text evidence is not enough to classify reliably.
Data pipeline

How the site is built

The public site is a static export of generated tables. The underlying code keeps the collection, enrichment, classification, validation, and trend steps auditable.

Data construction

metadata
01

Collect DOI-centered bibliographic metadata from Crossref and work metadata from OpenAlex.

02

Deduplicate records into an article-level panel using DOI, or normalized title-journal-year when DOI is missing.

03

Backfill missing text from cached abstracts, open metadata, and bounded PDF/OCR recovery when evidence is available.

Classification and trends

analysis
04

Classify articles into causal, predictive, other, or insufficient text using auditable rules and model-assisted labels.

05

Run validation diagnostics, overlap checks, adjudication, and a validation gate before reading trends.

06

Export trend tables into site/data.js, which powers this static GitHub Pages dashboard.

Interpretation boundary This is an exploratory descriptive dashboard. The current gate passes, but the latest validation pass is AI-assisted and should be reported that way unless a separate human audit subset is completed.

Sharing and replication

The repository should publish this static site through GitHub Actions and keep large raw/intermediate data outside git unless released as a separate replication archive.

Website

Deploy from site/

The GitHub Action packages the dashboard, selected public docs, and .nojekyll into a Pages artifact on every push to main.

Reproduction

Code first, data archived separately

Source code, configs, tests, prompts, and documentation belong in git. Large raw API caches, PDF caches, and generated output tables should be restored from an archived replication package when needed.
